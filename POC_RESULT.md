# PoC Result: zig-xaml-compiler (Subprocess Approach)

This report details the execution results of the Proof of Concept (PoC) for the `zig-xaml-compiler` project, focusing on the **Alternative B: Subprocess** approach (invoking `XamlCompiler.exe`).

## 1. Summary

The PoC successfully updated the `zig-xaml-compiler` to build and invoke the external `.NET 4.7.2` based `XamlCompiler.exe`. The Zig compilation of our tool succeeded cleanly using the Zig 0.15.2 API. However, the runtime execution of `XamlCompiler.exe` failed with `Exit Code: 1`, indicating that the `input.json` format or environment setup is still missing required properties.

## 2. Accomplishments

- **Redesign:** Shifted from the unstable FFI approach (`GenXbf.dll`) to the Subprocess approach (`XamlCompiler.exe`). Updated `DESIGN.md` accordingly.
- **Tool Discovery:** Located `XamlCompiler.exe` at `C:\Users\yuuji\.nuget\packages\microsoft.windowsappsdk\1.6.250108002\tools\net472\XamlCompiler.exe`.
- **Zig Implementation:** Successfully updated `build.zig` and `src/main.zig` to compile without errors under Zig 0.15.2. The tool now uses `std.process.Child` to spawn the compiler.
- **Build Success:** `zig build run` compiles the tool perfectly.

## 3. Issues Encountered (Runtime)

When running the spawned subprocess, `XamlCompiler.exe` returned **Exit Code: 1**.

- **Symptom:** `Compilation failed with exit code: 1`
- **Root Cause Analysis:** `XamlCompiler.exe` expects a highly specific and undocumented `input.json` format. While we provided `ReferenceAssemblies` (including `Windows.winmd` and `Microsoft.UI.Xaml.winmd`), `Language`, and `XamlPages`, it appears other undocumented MSBuild-injected fields (e.g., `TargetPlatformIdentifier`, `RootNamespace`, or structural differences in the JSON) are required to avoid immediate failure.
- **Error Visibility:** The standalone `XamlCompiler.exe` does not print standard error details to `stdout`/`stderr` when it fails on basic JSON validation or missing metadata, making debugging opaque.

## 4. Next Steps

To make the subprocess approach viable, the exact JSON structure generated by MSBuild must be captured:

1. **Capture the Real JSON:** Create a dummy C# WinUI 3 project in Visual Studio or via `dotnet new winui`. Build it, and intercept or inspect the `obj\` directory to find the actual `*.json` files passed to the `CompileXaml` MSBuild task.
2. **Reverse Engineer `input.json`:** Analyze the captured JSON to identify all strictly required fields, absolute paths, and array structures that `XamlCompiler.exe` demands.
3. **Update the Zig Tool:** Modify `src/main.zig` to generate this exact JSON structure dynamically before spawning the compiler.
4. **Implement PRI Generation:** Once `XamlCompiler.exe` succeeds and outputs an `.xbf` file, proceed with implementing the `MakePri.exe` subprocess call to index it into `resources.pri`.

## 6. Critical Discovery (diagnostic log analysis)

MSBuild diagnostic log から以下が判明:
- `XamlCompilerExePath` = `.../tools/net472/XamlCompiler.exe`
- `XamlCompilerJsonTaskPath` = `.../tools/net472/Microsoft.UI.Xaml.Markup.Compiler.IO.dll` — **JSON I/O を担当するDLL**
- `XamlCompilerTaskPath` = `.../tools/net472/Microsoft.UI.Xaml.Markup.Compiler.dll` — **コンパイラ本体のMSBuildタスクDLL**

**結論**: XamlCompiler.exe は単体で `input.json output.json` 引数で呼ぶのは正規の使い方ではない。`Microsoft.UI.Xaml.Markup.Compiler.IO.dll` が内部でJSONを組み立ててexeに渡す。

**次の選択肢**:
1. ILSpy で `Compiler.IO.dll` を逆アセンブルして正確なJSON構造を特定
2. MSBuild タスクとして正規に呼ぶ（`dotnet msbuild /t:CompileXaml` をZigから起動）
3. winui3-baseline のビルドでApp.xbfが生成されたので、そのxbfをそのまま使う（最小工数）
